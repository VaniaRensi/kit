# Refactor Engine — Systematic Code Rewriting

This skill translates audit findings into actual code changes. It runs after `core-audit` and before `core-quality` (final checkpoint).

**Two cardinal rules:**
1. **Originals are sacred** — never modify a file in the source project. Every rewritten file goes into the `[ID]-REFACTOR/` folder.
2. **Scope is sacred** — fix only what the audit identified, nothing else.

---

## 🗂️ The Refactor Sandbox

All work happens inside a dedicated mirror folder created at the project root before the first fix.

### Folder structure
```
[project-root]/
├── src/                        ← ORIGINAL — never touched, reference only
├── [ID]-REFACTOR/              ← SANDBOX — all new versions go here
│   └── src/
│       └── [only changed files, mirroring original paths]
├── AUDIT-PLAN.md               ← tracks fix progress
└── SWAP-GUIDE.md               ← generated as fixes complete (see below)
```

### Rules for the sandbox
- Mirror **only the files being changed** — do not copy the entire codebase.
- Preserve the **exact same relative path** as the original (`src/services/user.ts` → `[ID]-REFACTOR/src/services/user.ts`).
- The original stays runnable at all times. The sandbox is inert until the user runs the swap.

## 🚦 Pre-Flight Check (Before Any Fix)

Before the first change, confirm these are all true:

- [ ] `AUDIT-PLAN.md` exists and all 6 audit reports are STABLE.
- [ ] The user has confirmed: *"Proceed with refactor"* (do not self-start).
- [ ] `[ID]-REFACTOR/` folder has been created.
- [ ] `SWAP-GUIDE.md` has been initialized (see template below).
- [ ] `[ID]-STATUS.md` is updated: `OPERATION_MODE: BUILD`, `CURRENT_PHASE: PHASE 2`.

If any item is false, stop and resolve it before continuing.

### Initialize SWAP-GUIDE.md
Create this file at the project root before the first fix:

```markdown
# Swap Guide — [PROJECT_ID] Refactor
> Generated by core-refactor. Run these commands ONLY after reviewing and approving all files in [ID]-REFACTOR/.

## ⚠️ Before You Swap
- [ ] Review every file in `[ID]-REFACTOR/` against its original
- [ ] Run the project from the original to confirm it still works
- [ ] Get `core-quality` checkpoint approval

## Swap Commands
[filled in automatically as fixes complete — one line per file]

## After Swap
1. Delete `[ID]-REFACTOR/` folder
2. Run the project and verify
3. Commit with message: `refactor([scope]): apply audit-plan fixes`
```

## ⚙️ The Execution Loop (One Fix at a Time)

For **every item** in `AUDIT-PLAN.md`, in order CRITICAL → HIGH → MEDIUM → LOW:

### Step 1 — Declare
State the item explicitly before touching any file:

> "**Fix [N]/[total]** `[LEVEL]`: `[file/path.ts]`
> **Problem**: [what AUDIT-PLAN says is wrong]
> **Scope**: I will only touch [specific function / section / lines].
> **Standard**: loading `[skill-name]` for this fix type.
> **Verification plan**: [how I will confirm behavior is preserved]"

### Step 2 — Read
Read the **original** file as reference. Do not read the entire codebase — only what this fix needs.

### Step 3 — Write to Sandbox
Create the rewritten file at `[ID]-REFACTOR/[original/path/file.ext]`.
Rules:
- Fix **only** the identified problem. Do not opportunistically clean nearby code.
- If the fix reveals a new problem not in `AUDIT-PLAN.md`, add it as a new LOW item and continue.
- Do not mix a refactor with a feature addition in the same step.
- Load the skill from the Standards Reference table below before writing — the standard defines what "correct" looks like.

### Step 4 — Verify
Confirm the fix is correct by comparing the new file against the original:
- State explicitly what behavior is preserved.
- State what specifically changed and why it is better.
- If tests exist: describe how to run them against the new file.
- If no tests exist: describe the manual check the user should do.

### Step 5 — Update Records
Two updates, every time:

**`AUDIT-PLAN.md`** — check the item:
```
- [x] [LEVEL] `file/path.ts` — description ✅ Fixed [date]
```

**`SWAP-GUIDE.md`** — add the swap command for this file:
```bash
# Fix: [short description]
cp [ID]-REFACTOR/src/services/user.ts src/services/user.ts
```

### Step 6 — Checkpoint Gate (between levels)
After completing **all CRITICAL items**, STOP and run `core-quality` before moving to HIGH.
After completing **all HIGH items**, STOP and run `core-quality` before moving to MEDIUM.
MEDIUM and LOW can be batched into a single final checkpoint.

> "All CRITICAL fixes are complete and staged in `[ID]-REFACTOR/`. Running `core-quality` checkpoint before advancing to HIGH priority items."

## 📚 Standards Reference — Which Skill to Load Per Fix Type

Every fix belongs to a category. Before writing the new version in the sandbox, load the skill that defines the quality standard. You never work from memory — you read the standard.

| Fix Category | Skill to Load | What It Defines |
|---|---|---|
| Naming violations | `core-naming` | Conventions for variables, functions, files, DB columns |
| Architecture violations | `core-architecture` | Separation of concerns, feature-based structure, size limits |
| UI / Missing design tokens | `web-design-tokens` | 3-layer token system + token migration protocol |
| UI / Component patterns | `web-ux-ui` | Accessible primitives, layout systems, optimistic UI |
| UX / Hierarchy or accessibility | `core-ux` | 8 universal UX principles, audit checklist |
| Security vulnerabilities | `core-quality` §Security | RLS, env vars, sanitization, auth checks |
| Documentation gaps | `core-documentation` | What/how/when to document; README and interface standards |
| Architecture decision required | `core-architecture` | Stop refactor — redesign first, then return here |

## 🛡️ Scope Control Rules

| Situation | Rule |
|-----------|------|
| You spot an unrelated bug while fixing | Add it to `AUDIT-PLAN.md` as LOW. Do not fix now. |
| A fix requires touching a STABLE file | Warn the user. Proceed only with explicit approval. |
| The fix would change a public API/interface | Stop. Load `core-architecture` — redesign before refactoring. |
| A fix takes >30 lines of change | Split it into two items in `AUDIT-PLAN.md`. |
| User asks to add a feature mid-refactor | Decline. Features go after the refactor is STABLE and swapped. |
