# Audit Engine — Systematic Codebase Analysis

Use this skill to perform a professional-grade audit of an existing software project. The audit is non-destructive and results in a standardized documentation set plus one synthesized handoff file that drives the refactor and documentation phases.

**The audit produces 7 outputs, all saved inside a versioned `[ID]-AUDIT-v[N]/` folder at the project root. Nothing is written into the source code folders. Every time you run a new audit, a new version folder is created — old ones are preserved for comparison.**

```
[project-root]/
├── src/                      ← source code — never touched by the auditor
├── [ID]-AUDIT-v1/            ← first audit run
│   ├── AUDIT-MAP.md
│   ├── AUDIT-FLOW.md
│   ├── AUDIT-QUALITY.md
│   ├── AUDIT-SECURITY.md
│   ├── AUDIT-DEPS.md
│   ├── AUDIT-PLAN.md
│   └── PLAYBOOK.md           ← the 7th output; primary input for refactor and documentation
├── [ID]-AUDIT-v2/            ← second audit run (if re-audited after refactor)
│   └── ...
```

**Version detection**: before creating the folder, check for existing `[ID]-AUDIT-v*/` folders. Take the highest N and create `[ID]-AUDIT-v[N+1]/`. If none exist, start with `[ID]-AUDIT-v1/`.

## 🗺️ The 6 Reports

Every audit MUST produce these six files inside `[ID]-AUDIT/`. Write them in order — do not write `AUDIT-PLAN.md` until the other five are stable.

### 1. `AUDIT-MAP.md` (What exists?)
- **Goal**: Full inventory of the project.
- **Content**:
  - Tech Stack (Frameworks, DB, Auth).
  - Folder Structure Analysis (Tree).
  - Entry Points (Main files, API routes).

### 2. `AUDIT-FLOW.md` (How does data move?)
- **Goal**: Understand logic and state.
- **Content**:
  - Key User Flows (e.g., Auth flow, Payment flow).
  - Data Models (Tables/Schemas).
  - Component Hierarchy (Frontend).
  - State Management approach.

### 3. `AUDIT-QUALITY.md` (Code Health)
- **Goal**: Evaluate the codebase through the lens of every applicable Kit skill. Do not invent your own standards — read the relevant skill and apply its criteria.
- **Content**:
  - Naming: load `core-naming` — are conventions followed?
  - Architecture: load `core-architecture` — separation of concerns, file sizes, feature structure?
  - UI/UX: load `core-ux` — hierarchy, spacing, accessibility, heading capitalisation consistent throughout?
  - Tokens: load `web-design-tokens` — are design tokens used or are values hardcoded?
  - Performance Hotspots: visual bloat, heavy bundles, duplicate API calls.

### 4. `AUDIT-SECURITY.md` (Safety Scan)
- **Goal**: Identify critical vulnerabilities. Load `core-quality` §Security for the checklist.
- **Content**:
  - Env Var exposure check.
  - Auth/Permission logic verification.
  - Input Sanitization review.
  - SQL / NoSQL injection risks.

### 5. `AUDIT-DEPS.md` (Dependency Health)
- **Goal**: Surface risks from third-party packages.
- **Content**:
  - Run the appropriate scanner for the tech stack and record output:
    - **Node.js**: `npm audit --json`
    - **Python**: `pip-audit`
    - **Ruby**: `bundle audit`
    - **C# / .NET**: `dotnet list package --vulnerable` + `dotnet list package --outdated`
  - Outdated critical dependencies (flag anything >2 major versions behind).
  - Unused or duplicate dependencies (`depcheck` or equivalent).
  - License risks (GPL in a commercial project, etc.).
  - Lock-file status (is `package-lock.json` / `yarn.lock` committed?).

### 6. `AUDIT-PLAN.md` (Prioritized Action)
- **Goal**: Detailed fix list, synthesized from all 5 reports above.
- **Content**:
  - **CRITICAL**: Security vulnerabilities or broken functionality — fix immediately.
  - **HIGH**: Architecture violations, naming refactors, dependency CVEs.
  - **MEDIUM**: Feature additions, UX polish, outdated-but-stable deps.
  - **LOW**: Minor tweaks, optional improvements.
- **Format per item**:
  ```
  - [ ] [LEVEL] `file/path.ts` — short description of the problem
        Reason: why this is a risk or debt
        Fix: the specific change required
        Skill: [skill-name] — the kit skill that defines the correct standard
  ```

## 🚦 Audit Workflow

1. **Setup**: Detect the next version number (check existing `[ID]-AUDIT-v*/` folders, increment the highest N, default to v1). Create `[ID]-AUDIT-v[N]/`. All files from this point go inside it.
2. **Exploration**: Run `ls -R` and read the dependency manifest (`package.json`, `requirements.txt`, `*.csproj`, `Gemfile`, etc.). Then detect the language stack and load any matching `lang-*` skill:
   - `.csproj` or `.sln` found → load `kit/skills/lang-csharp/INSTRUCTIONS.md` before continuing.
   - `requirements.txt` or `pyproject.toml` found → check for `lang-python` skill.
   - `go.mod` found → check for `lang-go` skill.
   If a matching `lang-*` skill exists, load it now and apply its lens throughout the rest of the audit.
3. **Dependency Scan**: Run the appropriate scanner (see AUDIT-DEPS section above) and save raw output.
4. **Mapping**: Build `AUDIT-MAP.md` and `AUDIT-FLOW.md` inside the version folder.
5. **Deep Dive**: Sample 3–5 files per layer (UI, Service, Data). For each area, load the relevant Kit skill and evaluate the code against its standards — do not invent your own criteria.
6. **Reporting**: Finalize `AUDIT-QUALITY.md`, `AUDIT-SECURITY.md`, `AUDIT-DEPS.md` inside the version folder.
7. **Plan**: Write `AUDIT-PLAN.md` as a synthesis of the 5 reports above.
8. **Playbook**: Write `PLAYBOOK.md` from the template above. This is the handoff — once it exists, the audit is complete.
